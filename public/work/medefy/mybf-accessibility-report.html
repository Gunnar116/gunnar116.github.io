Official Findings
Summary of TestPros findings mapped to WCAG success criteria, design impact, and recommended remediation direction.
WCAG Criterion Android Finding iOS Finding Shared Recommendation
1.4.1
Use of Color
Failure
 Selected bottom nav tab turns blue with no secondary visual indicator — no underline, pill, or weight change to distinguish active state. iOS bottom tab bar uses the same brand blue tint for active state with no supplementary non-color differentiator on the selected tab. Add a non-color secondary cue: an active underline bar, pill indicator, bold label weight, or filled icon shape — in addition to the color change. Apply to both Android and iOS bottom navigation.
1.4.3
Contrast (Minimum)
Failure
 Secondary text, helper text, metadata, timestamps, placeholder, and inline links fail 4.5:1 on white backgrounds using current Android color tokens. iOS shares the same design token values — identical contrast failures appear on secondary labels, hint text, and link colors across iOS screens. Darken shared tokens: text/secondary → #61718F, text/placeholder → #676771, text/link → #0668E0. Update both Android colors.xml and iOS asset catalog / SwiftUI color sets.
1.4.11
Non-text Contrast
Failure
 Form field borders use #DDDDDD — fails 3:1 against white backgrounds on Invite Dependents and Contact Support screens. iOS form fields reference the same border token and exhibit identical contrast failure against white backgrounds — input boundaries are not visually distinct. Darken default field border to #949494 (3.9:1 on white). Apply focused ring in #0668E0 and error state in #C62828. Update both platforms.
⚠
Audit Scope
These findings were originally identified by TestPros on Android. Subsequent review confirmed that iOS shares the same design tokens and exhibits the same failures. All remediation recommendations in this document apply to both Android and iOS. Web surfaces using the same tokens should be evaluated separately.

Implementation Note
These token values should be updated in Android resource files (colors.xml) and iOS asset catalogs (Assets.xcassets) or SwiftUI color sets, as well as in any design token source (Figma variables, style dictionary). Error State Accessibility
Error messages must use role="alert" or aria-describedby so screen readers announce them — color alone is never sufficient.
iOS border note: iOS UITextField uses a system border style by default. For custom-bordered fields, ensure layer.borderColor is set to UIColor(hex: "#949494") in default state. SwiftUI .overlay(RoundedRectangle...) borders should use the same token.
iOS implementation note: Stroke visibility on iOS retina displays can make 1pt borders appear thinner than on Android. Test field borders on physical devices at 2× and 3× scale to confirm #949494 remains perceptually distinct from white backgrounds.

Cross-Platform Recommendation
All color token updates documented in this report apply equally to Android and iOS. Both platforms share the same design token source and exhibit the same WCAG failures. The remediation strategy is intentionally unified: a single updated token set should be published to both platforms simultaneously, rather than managing separate Android and iOS palettes. This approach reduces long-term maintenance burden, prevents platform divergence, and ensures a consistent accessible experience for all users regardless of device. After implementation, conduct accessibility audits on physical devices of both platforms to validate contrast ratios in real rendering conditions.
